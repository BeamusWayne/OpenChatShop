# 测试策略

## 测试层次

### 单元测试（必须）

每个功能的核心逻辑必须有单元测试：
- 纯函数：直接测试输入输出
- 有副作用的函数：mock 外部依赖，测试内部逻辑
- 边界条件：空值、错误输入、极端情况

验证命令：使用 `.harness/config.json` 中的 `verify` 命令。

### 集成测试（推荐）

功能涉及多个模块交互时：
- API 端点：测试请求/响应，包括错误情况
- 数据库操作：测试 CRUD，包括约束和冲突
- 外部服务：mock 或 stub，测试调用和错误处理

### 端到端测试（按需）

关键用户流程需要端到端验证：
- 用户注册 → 登录 → 执行操作 → 退出
- 核心业务流程的完整路径

## 测试优先级

1. **基础设施功能**（feat-001）必须包含测试框架配置和至少一个冒烟测试
2. **业务功能**的核心逻辑必须有单元测试
3. **API 端点**必须有请求/响应测试
4. **UI 交互**按需做端到端测试

## 验证证据

在 feature_list.json 的 evidence 字段中记录：

```
## 测试证据
- 测试命令：<运行的具体命令>
- 测试结果：<通过/失败，X passed, Y failed>
- 覆盖范围：<测试了哪些模块/函数>
- 未覆盖：<哪些路径没有测试，为什么>
- 手动验证：<手动检查了什么行为>
```

## 常见测试框架

| 语言 | 框架 | 命令 |
|------|------|------|
| JavaScript/TypeScript | Jest / Vitest | `npm test` / `npx vitest` |
| Python | pytest | `pytest` |
| Go | testing | `go test ./...` |
| Rust | cargo test | `cargo test` |
| Bash | bats | `bats tests/` |

## 测试原则

- **测试意图，不是实现**：测试"注册成功后应该收到确认邮件"，不是"调用 sendMail 函数"
- **每个测试一个断言主题**：一个测试只验证一个行为
- **测试名称描述场景**：`should reject duplicate email on registration` 而不是 `test1`
- **不为了覆盖率写测试**：80% 覆盖率是底线，但 100% 无意义的测试不如 50% 有意义的测试
