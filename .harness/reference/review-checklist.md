# 代码审查检查单

功能标记 passing 前，必须逐项检查并记录具体发现。不是打勾就走，而是写明扫描了什么、发现了什么。

## 安全性

- **硬编码密钥/密码**：扫描所有新增文件，确认无 API key、密码、token、secret。记录：扫描了哪些文件（文件名:行号），有无发现。
- **用户输入验证**：列出所有外部输入入口（API 端点、命令行参数、文件读取），确认每个都有验证。记录：哪些入口（文件名:行号），验证了什么。
- **注入风险**：检查 SQL 查询是否用参数化、HTML 输出是否转义、命令是否拼接。记录：检查了哪些查询/输出（文件名:行号）。
- **权限控制**：如果有认证相关功能，确认权限检查存在。记录：检查了哪些路由/操作（文件名:行号）。

## 正确性

- **边界条件**：列出功能涉及的所有边界（空值、零值、最大值、重复、并发）。记录：哪些边界（文件名:行号），如何处理。
- **错误处理**：检查所有可能失败的路径（网络、文件、数据库、解析）。记录：哪些路径（文件名:行号），错误如何传播。
- **数据一致性**：如果有状态变更，确认操作的原子性和一致性。记录：哪些状态变更（文件名:行号），是否有事务/锁保护。

## 可维护性

- **函数大小**：新增函数是否 < 50 行？如果不是，指出哪个函数超了（文件名:行号）。
- **文件大小**：修改后的文件是否 < 800 行？如果不是，是否需要拆分（文件名:当前行数）？
- **嵌套深度**：是否有 > 4 层嵌套？如果有，指出位置（文件名:行号）。
- **命名**：新标识符是否清晰表达意图？有没有需要改名的（文件名:行号:标识符名）？
- **重复代码**：新增代码是否与已有代码重复？如果有，是否应该提取（文件名:行号）？

## 测试

- **关键路径覆盖**：核心业务逻辑是否有测试？记录：哪些路径有测试（测试文件名:行号），哪些没有（源文件名:行号）。
- **测试意图**：测试是否验证"为什么"而不仅是"做了什么"？记录：哪些测试有明确的意图说明（测试文件名:行号）。
- **无跳过测试**：是否有 skipped/ignored 的测试？如果有，为什么（测试文件名:行号）？
- **验证命令**：测试是否通过？运行了什么命令，结果如何？

## 审查结果模板

在 feature_list.json 的 evidence 字段中记录：

```
## 代码审查
- 安全：[扫描了 X 个文件（文件名:行号），发现/未发现 Y]
- 正确性：[检查了 X 个边界条件（文件名:行号），发现/未发现 Y]
- 可维护性：[X 个新函数（文件名:行号），最大 Y 行；文件大小 Z 行]
- 测试：[X 个新测试（测试文件名:行号），覆盖 Y 个路径；运行结果：通过/失败]
- 结论：[通过/需要修复 X]
```
