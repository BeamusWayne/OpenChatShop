# 规划方法论

## 目标

把一个功能拆解成可独立执行、可验证的原子任务，让编码过程有据可循。

## 拆解原则

1. **每个任务 2-5 分钟可完成** — 如果一个任务描述超过 3 行，说明还要拆。
2. **每个任务有明确的输入和输出** — 知道从哪开始，到哪结束。
3. **每个任务有验证命令** — 能判断"完成了"还是"没完成"。
4. **任务之间有依赖但尽量独立** — 标注前置任务，但允许并行。

## 拆解步骤

### 第 1 步：列出文件清单

对当前功能，列出需要创建或修改的文件：

```
创建：src/auth/register.ts
创建：src/auth/register.test.ts
修改：src/routes/index.ts（添加注册路由）
修改：src/db/schema.ts（添加用户表）
```

### 第 2 步：为每个文件描述改动

```
src/db/schema.ts:
  - 添加 users 表定义（id, email, password_hash, created_at）
  - 导出 users 类型

src/auth/register.ts:
  - 导出 registerUser(email, password) 函数
  - 输入验证：邮箱格式、密码长度
  - 密码哈希（bcrypt）
  - 数据库写入
  - 返回用户 ID 或错误

src/auth/register.test.ts:
  - 测试：有效输入成功注册
  - 测试：重复邮箱返回错误
  - 测试：无效邮箱格式被拒绝
  - 测试：短密码被拒绝

src/routes/index.ts:
  - POST /auth/register 路由
  - 调用 registerUser
  - 返回 201 或 400
```

### 第 3 步：排序任务

按依赖关系排序：

1. 数据库 schema（无依赖）
2. 注册函数实现（依赖 schema）
3. 注册函数测试（依赖实现）
4. 路由接入（依赖实现）

### 第 4 步：写出验证步骤

每个任务的验证命令：

```
任务 1: 无（schema 变更在迁移时验证）
任务 2: 无独立验证（被任务 3 覆盖）
任务 3: npm test -- src/auth/register.test.ts
任务 4: npm test（全量测试）
```

## 计划文件格式

用 `harness new-plan <feature-id>` 创建计划后，填充以下结构：

```markdown
# 执行计划：<功能标题>

## 文件清单
| 文件 | 操作 | 改动描述 |
|------|------|----------|
| ...  | 创建/修改 | ... |

## 任务列表
| # | 任务 | 验证命令 | 依赖 |
|---|------|----------|------|
| 1 | ...  | ...      | -    |
| 2 | ...  | ...      | 1    |

## 风险
- <潜在问题>：<缓解措施>
```

## 常见错误

- **跳过计划直接编码** — 导致返工、遗漏边界条件
- **任务粒度太粗** — "实现用户注册"不是一个任务，是多个
- **没有验证命令** — 无法判断任务是否完成
- **忽略依赖关系** — 先写路由再写实现，编译失败
