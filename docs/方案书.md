# 电商智能对话系统 — 开源项目方案书

> 项目代号：**OpenChatShop**
> 版本：v0.3
> 日期：2026-05-19

---

## 一、项目概述

构建一个**模型无关（Model-Agnostic）**的开源电商智能对话系统。企业可选择任意 LLM 后端（Claude、GPT、Qwen、DeepSeek、本地部署模型等），快速搭建生产级电商客服 Agent。

核心目标：模型无关、可插拔业务模块、生产就绪、开发者友好、多渠道覆盖（Web / APP / 微信）。

## 二、设计哲学

**"LLM 只负责编排，确定性代码负责执行"**

- LLM 输出意图 + 参数，不做数据库操作
- 写操作由确定性代码执行，经过校验、确认、审计
- 状态机转换由代码驱动，LLM 只建议下一步
- 工具参数由代码校验和过滤

## 三、系统架构

```
多渠道接入层 → Gateway → 安全防护层 → Agent 编排引擎 → 数据服务层 → 可观测层
```

**编排引擎核心流程：** 安全检测 → 加载上下文 → 意图识别 → 动态注入工具 → 策略决策 → 执行动作 → 更新上下文

详细架构图和数据流见 [contracts.md](./design/contracts.md) §8。

## 四、模块概览

每个模块有独立设计文档，接口定义统一在 [contracts.md](./design/contracts.md) 中。

### 4.1 Provider 抽象层

统一接口适配多家 LLM，配置驱动切换。三级级联策略：规则 + 语义检索（零成本）→ 通用 LLM → 高精度 LLM，80% 请求在第一级解决。

详见 [provider-layer.md](./design/provider-layer.md)

### 4.2 上下文管理器

滑动窗口 + 摘要压缩 + 实体槽位追踪。Token 预算按比例分配：System Prompt 20%、对话历史 50%、工具结果 20%、槽位实体 10%。

详见 [context-manager.md](./design/context-manager.md)

### 4.3 意图识别引擎

三级级联：规则匹配 → 语义向量检索 → LLM 分类。每级有置信度阈值，不满足则升级到下一级。结合上下文历史和 FSM 状态辅助消歧。

详见 [intent-engine.md](./design/intent-engine.md)

### 4.4 工具系统

意图驱动动态注入（非全量加载），经意图匹配、场景过滤、权限过滤、数量截断四层筛选。写操作需确认机制和补偿逻辑。安全矩阵控制不同操作的权限级别。

详见 [tool-system.md](./design/tool-system.md)

### 4.5 对话体验

以 Web 为首要交付形态（嵌入式 Chat Widget），通过 Channel Adapter 支持微信公众号、小程序、APP 内嵌。11 种消息类型（文本、商品卡片、订单卡片、物流时间线、确认弹窗等），不支持的消息类型自动降级。

详见 [conversation-ux.md](./design/conversation-ux.md)

### 4.6 数据架构

三层存储：向量库（Milvus/pgvector/ChromaDB 可插拔）+ 关系数据库（PostgreSQL，按月分区、读写分离）+ Redis Cluster 缓存（会话/热数据/限流）。通过 Debezium CDC 同步。

详见 [data-architecture.md](./design/data-architecture.md)

### 4.7 安全防护

四层防护链：Prompt 注入检测（规则 + ML 双引擎）→ 内容安全过滤（PII 检测 + 敏感词）→ 工具调用权限校验（RBAC）→ 输出脱敏。认证支持 JWT / API Key / 微信 OAuth。

详见 [security.md](./design/security.md)

### 4.8 评测体系

黄金数据集（500+ 标注样本）+ CI 自动回归测试 + LLM-as-Judge 评分 + A/B 测试框架。目标：意图准确率 >= 95%，工具调用成功率 >= 99%，P99 <= 2s。

详见 [evaluation.md](./design/evaluation.md)

### 4.9 部署与成本治理

Docker Compose 开发环境 / K8s Helm 生产部署 / Ollama 私有化部署。Token 成本通过级联策略控制，单会话预算硬限制，成本监控看板和告警。

详见 [deployment.md](./design/deployment.md)

## 五、技术选型

| 层次 | 技术 | 说明 |
|------|------|------|
| 语言 | Python 3.11+ | async/await 原生支持 |
| Web 框架 | FastAPI | 异步、Pydantic 原生集成 |
| LLM 集成 | LiteLLM + 薄封装 | LiteLLM 处理 80% 适配；自研级联策略与降级逻辑 |
| ORM | SQLModel | FastAPI 亲儿子，Pydantic + SQLAlchemy 融合 |
| 关系数据库 | PostgreSQL 15+ | 按月分区、读写分离 |
| 向量数据库 | pgvector（Phase 1）→ Milvus（生产） | Phase 1 与 PG 共用一个实例，减少初始依赖 |
| 搜索引擎 | Elasticsearch（可选） | 商品搜索等场景 |
| 缓存 | Redis | 会话/热数据/限流；Phase 1 单节点 |
| 消息队列 | Redis Streams | Phase 1 不引入独立 MQ |
| 数据同步 | Debezium CDC | Phase 2+ |
| 测试 | pytest + pytest-asyncio | FastAPI 生态标准 |
| 可观测 | OpenTelemetry + Grafana | 统一 tracing/metrics/logs |
| 容器化 | Docker + Docker Compose + K8s Helm | 开发用 Compose，生产用 Helm |
| 前端 SDK | React + WebSocket | 嵌入式 Chat Widget |

## 六、项目结构

```
open-chat-shop/
├── src/
│   ├── core/           # 核心引擎（provider/context/intent/strategy/tool/security/auth）
│   ├── tools/          # 内置电商工具集
│   ├── storage/        # 存储适配层（session/vector/knowledge/audit）
│   ├── channel/        # 多渠道适配层（web/wechat/app）
│   ├── api/            # API 层（rest/websocket/admin）
│   ├── observability/  # 可观测性 + 成本追踪
│   └── evaluation/     # 评测框架
├── sdk/web/            # Web Chat Widget（React）
├── sdk/mobile/         # 移动端 SDK（React Native）
├── configs/            # 配置文件（providers/tools/security/scenarios/channels）
├── tests/              # unit/integration/e2e
├── docs/               # 文档（含 design/ 子目录）
├── examples/           # 示例（含 quick_start.py）
└── data/seed/          # 种子数据
```

## 七、实施路线图

### Phase 0：Quick Start Demo（第 0 周，1-2 天）

单文件 `quick_start.py` + Ollama 本地模型 + Mock 工具，5 分钟跑起来，零外部依赖。

### Phase 1：核心框架（第 1-4 周）

Provider 抽象层、上下文管理器、基础对话 API、Docker Compose、Web Chat Widget v1。

### Phase 2：Agent 能力（第 5-8 周）

意图识别引擎、工具系统（动态注入）、状态机编排、电商工具集、安全防护、多渠道适配、富消息组件。

### Phase 3：生产增强（第 9-12 周）

人机协作、可观测性、评测框架、成本治理、K8s Helm、更多 Provider。

### Phase 4：生态建设（第 13-16 周）

管理后台、工具市场、Provider SDK、APP SDK、性能基准、A/B 测试、i18n。

## 八、竞品分析

| 项目 | 模型依赖 | 电商场景 | 工具调用 | 多渠道 |
|------|---------|---------|---------|--------|
| **OpenChatShop** | **模型无关** | 深度电商 | 动态注入 + 安全矩阵 | 5 渠道 |
| Dify | 模型无关 | 通用 | 静态注册 | Web + API |
| Rasa | 模型无关 | 需自建 | 无内置 | 需自建 |
| FastGPT | OpenAI 为主 | 通用 | 基础 | Web |
| Coze | 豆包绑定 | 通用 | 丰富 | 多渠道（非开源） |

**差异化：** 级联降本、意图驱动工具注入、"LLM 只编排"原则、电商深度场景 FSM、成本内建、多渠道降级适配。

## 九、开放治理

- **许可证：** Apache 2.0
- **贡献类型：** Bug 修复 / 新 Provider / 新工具 / 新场景 FSM / 新渠道适配器 / 文档
- **社区：** GitHub Discussions、双周 Community Call、季度 Roadmap Review

## 十、风险与应对

| 风险 | 应对 |
|------|------|
| LLM API 不稳定 | 多 Provider Fallback + 本地模型兜底 |
| Prompt 注入攻击 | 四层安全防护 + 红队测试 |
| 工具调用幻觉 | 参数校验 + 确认机制 + 权限矩阵 |
| 上下文窗口不足 | Token 预算管理 + 动态压缩 |
| Token 成本失控 | 级联策略 + 预算硬限制 + 成本告警 |
| 多渠道碎片化 | 统一消息协议 + Channel Adapter 降级 |
| 数据安全合规 | 私有化部署 + 数据脱敏 + 审计日志 |
| 社区参与度低 | 降低贡献门槛 + 完善文档 + 快速响应 PR |
