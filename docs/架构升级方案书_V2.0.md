# OpenChatShop 架构升级改进方案书 (V2.0)

**日期**: 2026-06-05
**主题**: 从单体智能客服向企业级 Multi-Agent 协作架构演进

---

## 1. 核心目标
当前的 `open_chat_shop` 已具备完善的工具调用、安全拦截和人机协同能力，底层工程基建（测试、监控、限流）极其扎实。
下一阶段的演进目标是：**降低单模型上下文负担、提高长尾复杂意图的准确率、实现千人千面的个性化服务体验，并根除安全护栏的误杀问题。**

## 2. 重点改进模块与架构设计

### 2.1 模块一：从“单体 Orchestrator”到“网关路由 + 专家智能体 (Multi-Agent)”
**问题痛点**：目前所有的意图分类、工具调用决策和话术生成都由一个核心的 `DialogueOrchestrator` 配合一个全局 System Prompt 完成。随着电商工具（物流、退款、推荐、发票）的增加，Prompt 膨胀会导致大模型能力退化和幻觉增加。

**解决方案：分层智能体架构**
*   **Triage Agent（意图路由网关）**：
    *   **职责**：仅负责分析用户的初始意图、情绪和紧迫性。
    *   **实现**：使用低延迟、低成本模型（如 Flash 版本模型）或轻量级本地分类模型。
    *   **动作**：将对话路由给特定的下游专家 Agent，或在检测到极高负面情绪时直接触发 `handoff`。
*   **Domain Specialist Agents（领域专家）**：
    *   **职责**：拆分为 `Refund Agent`（售后专家）、`Sales Agent`（导购专家）、`Logistics Agent`（物流专家）。
    *   **实现**：每个 Agent 仅携带与自身领域相关的 API Tool 和专属 Prompt。例如，导购专家不需要知道如何调用退款接口。
*   **收益**：Token 消耗降低约 30%，工具调用错误率大幅下降，业务逻辑解耦，易于扩展新业务线。

### 2.2 模块二：基于 `pgvector` 的生产级混合检索 (Hybrid RAG)
**问题痛点**：当前的语义检索层 (`InMemoryVectorStore`) 采用全表线性扫描计算余弦相似度，算法复杂度为 `O(N log N)`。一旦导入海量电商 FAQ、活动规则或退换货政策，将成为巨大的性能瓶颈。

**解决方案：引入向量数据库与重排序**
*   **向量化存储**：利用项目中已存在的 `pgvector` 可选依赖，将 SQLite/InMemory 迁移到 PostgreSQL。为电商商品数据和非结构化规则（FAQ）建立 HNSW 向量索引。
*   **Hybrid RAG**：
    *   结构化查询（如查订单状态）继续走现有的 Function Calling。
    *   非结构化查询（如“这款衣服缩水吗？”）走 RAG 检索商品详情或 FAQ 库。
*   **重排序 (Re-ranker)**：在初步召回 Top-10 相关内容后，引入轻量级 Re-rank 模型提取最精准的 Top-3 喂给 LLM，减少无关信息的干扰。
*   **收益**：解决大体量知识库下的检索性能悬崖，大幅降低回答产品特定问题时的幻觉。

### 2.3 模块三：长期记忆 (Long-Term User Persona) 与个性化服务
**问题痛点**：现有的 `ContextManager` 虽然能在单次会话（Session）中维持记忆，但缺乏跨 Session 的长期记忆体系，无法建立用户画像。

**解决方案：异步画像提取流水线**
*   **Memory Extraction**：在对话结束（或用户长时间不活跃）后，通过异步队列触发一个提取任务，利用小模型总结用户的核心特征。
    *   *示例提取内容*：“用户尺码偏好 L”、“对价格敏感”、“喜欢日系极简风”、“家中有宠物”。
*   **Persona Storage**：将会话摘要存入用户维度（基于 `user_id`）的画像表或图数据库中。
*   **Context Injection**：当用户再次发起对话时，Triage Agent 自动检索该用户的画像标签，并静默注入到下游专家的 System Prompt 中。
*   **收益**：实现“千人千面”的服务体验（如：“按您上次的购买习惯，我给您推荐这一款...”），提升复购率和客服满意度。

### 2.4 模块四：语义护栏体系升级 (Semantic Guardrails)
**问题痛点**：目前的 `SecurityGuard` 中 `_check_special_char_ratio` 等硬编码正则规则过于粗暴，导致用户发送含有多个 Emoji 或中文标点的正常短句被误判为恶意注入，严重影响真实用户体验。

**解决方案：基于 LLM 的轻量级护栏引擎**
*   **Input Guard（输入净化）**：
    *   移除僵硬的特殊字符比例硬编码。
    *   引入基于本地极小模型（或专门微调的文本分类模型）进行 Prompt Injection 和 Toxic Content 识别。
*   **Output Grounding Check（输出幻觉校验）**：
    *   **核心逻辑**：对涉及敏感数值（退款金额、价格、订单状态）的回复进行“反向校验”。在将回复发给用户前，增加一道程序化断言，确保 LLM 话术中提及的数字与 `tool` 返回的结果严格一致。
*   **收益**：彻底消除 Emoji 和正常语气的误报；阻断大模型“凭空发钱”导致的安全生产事故。

## 3. 落地实施路径与阶段规划

| 阶段 | 核心任务 | 预期周期 | 交付物 |
| :--- | :--- | :--- | :--- |
| **Phase 1: 护栏重构与知识库升级** | 1. 废除硬编码规则，引入轻量级分类或模型输入护栏。<br>2. 接入 `pgvector`，实现高维索引检索。 | 1 周 | 无误杀的 SecurityGuard；基于 Postgres 的 VectorStore。 |
| **Phase 2: 架构解耦与路由** | 1. 重构 Orchestrator，剥离出 Triage Router。<br>2. 封装独立的 Domain Agent 基类并迁移现有 Tool。 | 2 周 | Multi-Agent 协同引擎；业务解耦的各领域 Specialist。 |
| **Phase 3: 记忆与主动服务** | 1. 开发异步画像提取服务。<br>2. 实现上下文注入机制与长效记忆联动。 | 2 周 | User Persona 存储模型；千人千面对话能力。 |

## 4. 结论
通过本次架构升级，`open_chat_shop` 将从一个基础的“工具调用型机器人”蜕变为符合业界最新标准的 **企业级 Multi-Agent 协作网络**。这不仅能解决性能与上下文膨胀的瓶颈，更能通过长期记忆和语义护栏，提供极致安全且极具温度的电商客户体验。
